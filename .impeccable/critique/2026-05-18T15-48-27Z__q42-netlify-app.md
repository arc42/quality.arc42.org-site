# Impeccable Critique: q42 live site

Target: https://q42.netlify.app/
Date: 2026-05-18

Verdict: Ship-worthy and useful. The site now feels like a real professional reference tool, not a generic AI landing page. The graph, domain language, dense lookup pages, and Cmd-K search all support the intended audience. It is not yet impeccable because mobile navigation, graph usability, search empty state, and visual-system consistency still leak friction.

Inspected live routes: home, qualities, requirements, approaches, standards, quality detail, requirement detail, search, full graph. Desktop and mobile screenshots were captured. Sampled pages returned HTTP 200, no console errors, no failed requests, and no horizontal overflow.

## Anti-Patterns Verdict

AI slop verdict: Pass. I would not immediately call this AI-generated. The purple hero, reference-tool density, Atkinson typography, and graph give it a specific identity.

Remaining tells: repeated pastel bordered panels, accent strips, large serif headings everywhere, and mixed card/list systems sometimes feel assembled rather than resolved. No gradient text, no glassmorphism, no hero metric template, no generic SaaS hero.

Static checks found remaining side-stripe accent rules in source, including _sass/_aliases.scss, _sass/_standards.scss, _sass/_content.scss, and _pages/20-quality-characteristics.md. Within the /impeccable rubric this is a banned pattern. It is not an AGENTS.md repository rule.

## What Works

The homepage now communicates the model quickly, with corrected counts and a credible first impression.

Search is a major win. Typing availability opens a focused result, Enter navigates directly, and mobile search returns grouped results.

The qualities explorer is genuinely useful: dimensions, letter jumps, alias accounting, and result totals make a large taxonomy navigable.

## Priority Issues

[P1] Mobile header and navigation consume too much of the first viewport.
Why it matters: on a 390px screen, users see logo, all primary links, search, and a hamburger before content. The hamburger then reveals duplicated primary links plus more links. This creates navigation weight before the actual reference task starts.
Fix: choose one mobile nav model. Either keep visible primary links and remove the hamburger, or collapse primary links into the hamburger and make search the only persistent control.
Suggested command: $impeccable adapt

[P1] The search results page has a broken empty/no-query state.
Why it matters: /search/ currently shows Search Results plus Initializing search... and then appears done. This undermines trust in a core lookup workflow.
Fix: show an empty state with the search input, suggested queries, or parse q= and render results. Never leave a non-loading page saying Initializing.
Suggested command: $impeccable harden

[P2] The full graph is impressive on desktop but weak on mobile.
Why it matters: the graph is the signature feature, but on mobile the first render is tiny and right-shifted, with labels too small to be useful. It proves the graph exists, but does not yet make it productive.
Fix: center the graph after layout settles, add a mobile-first graph mode with larger primary dimension nodes, and offer a list fallback after filtering.
Suggested command: $impeccable adapt

[P2] Page families still feel visually uneven.
Why it matters: qualities, standards, approaches, requirements, and detail pages use related but different card/panel/link treatments. This makes the site feel improved in places rather than fully systematized.
Fix: standardize hero cards, mode selectors, chips, result rows, citation boxes, and detail sections across page families.
Suggested command: $impeccable polish

[P3] The requirements page still looks older than the rest.
Why it matters: red underlined links and target bullets read like legacy content styling beside the more polished qualities, standards, and approaches explorers.
Fix: move requirements to the same explorer/list vocabulary: chips, grouped rows, quieter link styling, and consistent section headers.
Suggested command: $impeccable layout

## Persona Red Flags

First-time architect: the homepage works, but the qualities page immediately exposes 217 visible terms. They need strong search/filter primacy before the complete list.

Workshop facilitator: search and direct detail pages are strong, but inconsistent requirement styling may reduce confidence when projecting or using the site in front of a team.

Mobile reader: the header and nav take too much space, and the full graph is visually present but not comfortably usable.

## Minor Observations

The footer is clean on desktop but dominates mobile end states, especially the INNOQ block.

The Cmd-K hint is useful on desktop, but less relevant on mobile. The slash hint on the graph page is also desktop-biased.

The standards page cards are good, but the By category and By explorer controls look like cards rather than an active mode switch.

## Questions

Should the next pass focus on mobile navigation and graph usability, or on unifying the page-family visual system?
