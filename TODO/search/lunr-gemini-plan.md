# Plan: Migration to Lunr.js Search

## Status Quo Analysis
Currently, the site uses a manual regex-based search:
1.  **Index**: A `search.json` file (~1.5MB - 3MB estimated) is generated by Jekyll, containing the full text of all qualities, requirements, standards, and approaches.
2.  **Runtime**: The browser downloads the full `search.json` and performs a sequential regex scan across all entries.
3.  **UI**: The `search.html` page pre-renders *every single page* in the collections as a hidden `div`. The JS simply toggles visibility.

### Bottlenecks
- **Initial Load**: `search.html` is huge because it contains the HTML for 400+ articles.
- **Search Performance**: Regex matching on large strings in a loop is $O(N \times M)$ and doesn't support fuzzy matching, stemming, or field weighting.
- **Memory**: Keeping 400+ pre-rendered articles in the DOM is memory-intensive.

## Proposed Architecture: Lunr.js

### 1. Data & Indexing
- **Consolidate Synonyms**: Standardize on `aka:` (already used in qualities) and potentially `alias:` for all collections.
- **Node.js Build Integration**: Instead of generating a raw JSON of all content and indexing in the browser, we will pre-build the Lunr index during the JavaScript build pipeline.
    - **Implementation**: Add a Node.js script `src/scripts/index-search.js` and call it from `src/scripts/build.js`.
    - **Dev Workflow**: Since `make dev` runs `npm run watch`, any change to content or code will automatically trigger a re-index, keeping search results fresh in real-time.
    - **Fields**: 
        - `title` (Boost: 10)
        - `aka` / `alias` (Boost: 5)
        - `tags` (Boost: 3)
        - `content` (Boost: 1)
- **Content Stripping**: The index should only contain the searchable text. A separate `search-data.json` (or the existing `search.json` but optimized) will store the display metadata (title, URL, snippet).

### 2. Runtime Implementation
- **Lazy Loading**: Load `lunr.min.js` and the `search-index.json` only when the user interacts with search.
- **Dynamic Rendering**: Remove the pre-rendered hidden divs in `search.html`. Use a JavaScript template (or simple DOM manipulation) to render search results dynamically from the metadata.
- **Fuzzy Matching**: Enable Lunr's built-in fuzzy matching to handle typos.

### 3. Impact Assessment
- **Load Time**: 
    - `search.html` will shrink from several megabytes to a few kilobytes.
    - First search might have a slight delay while downloading the index (~200KB - 500KB compressed).
- **Runtime**:
    - Search will be near-instant ($O(1)$ or $O(\log N)$ depending on Lunr's implementation) regardless of the number of items.
    - Memory usage will drop significantly as we only render the top ~10-20 results.

## Implementation Steps

### Phase 1: Preparation (Infrastructure)
1.  Add `lunr` to `package.json`.
2.  Create `src/scripts/index-search.js` to build the Lunr index.
3.  Modify `src/scripts/build.js` to run the indexer.

### Phase 2: Metadata & Aliases
1.  Verify all collections support `aka` or `alias`.
2.  Update `search.json` template to include these fields.

### Phase 3: Frontend Overhaul
1.  Rewrite `assets/js/search.js` to use Lunr.
2.  Modify `_pages/search.html` to remove pre-rendered content and add a results container.
3.  Implement a result template (Title, Category, Snippet).

### Phase 4: Optimization
1.  Implement "search as you type" (debounce).
2.  Add result highlighting.
