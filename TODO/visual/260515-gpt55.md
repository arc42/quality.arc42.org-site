# GPT-5.5 visual critique - q42 violet redesign - 2026-05-15

Scope: live site at <https://q42.netlify.app/> and source branch `418-violet-redesign`.

Audience lens: software architects, dev teams, consultants, trainers, requirements engineers. This is a working reference surface, not a sales site. The most important qualities are fast lookup, information reliability, source confidence, and predictable cross-navigation.

## Method

- Inspected live pages: `/`, `/qualities/`, `/qualities/accessibility`, `/tag-usable/`, `/tag-reliable/`, `/dimensions/`, `/requirements/accessible-user-interface`, `/requirements/`, `/approaches/`, `/standards/`.

- Detector result: `[]`, no static pattern findings.

## Executive verdict

The redesign has moved in the right direction. The homepage, section overview heroes, autocomplete search, and requirement detail breadcrumbs now feel much more like a serious technical reference. The current weak point is no longer "generic SaaS". It is system consistency: new quiet editorial components and old saturated metamodel panels now coexist on the same routes.

The biggest visual issue is exactly the one noted in the prompt: `/qualities/` uses the new paper-card section hero, but single quality pages still use a full cyan slab. Different is fine. Unrelated is not. The detail header should remain distinguishable, but it should clearly belong to the same component family.

Tag pages are also close but underpowered. They use the dark dimension blue only as a narrow rail, while the page content then drops back into legacy Markdown and long lists. Since tags are quality dimensions in the metamodel, they should feel like a blue/navy "quality heading" surface, not a neutral filter page.

## Recent improvements worth keeping

- Header search is now genuinely useful: `Cmd+K`/`/`, grouped results, keyboard active row, accessible combobox attributes, and entity color rails.
- Requirement detail pages now have breadcrumbs and a capped related-requirements section. This fixes a major previous cognitive-load issue.
- Index pages now share `section-hero`, which gives `/qualities/`, `/requirements/`, `/approaches/`, `/standards/`, and tag pages a calmer common language.
- Standards overview no longer reads as a flashy campaign surface. The current card grid is still a little large, but it is not AI-slop.

## Anti-pattern verdict

This does not look like a generic AI-generated interface. The palette is specific, the typography feels like a handbook, and the homepage avoids SaaS cliches.

The remaining pattern smell is "two eras of UI stitched together":

- New quiet paper-card heroes: `_sass/components/_section-hero.scss:9`.
- Old saturated entity headers: `_includes/quality-header.liquid:4`, `_sass/components/_panels.scss:1`.
- Old full-width section bars: `_sass/components/_section-headings.scss:1`.
- Legacy tag-page include blocks with inline styles and repeated ids: `_includes/one-quality.liquid:10`, `_includes/one-requirement.liquid:5`.

That inconsistency matters more for this site than decorative polish. Dev teams infer reliability from repeatable structure.

## Prioritized todo list

### P1 - Unify overview and detail header vocabulary

Problem: `/qualities/` uses the new `section-hero` paper card with a blue rail. `/qualities/accessibility` uses a saturated cyan slab with a very large title and a separate pill. The relationship is conceptually correct but visually too abrupt.

Why it matters: a quality detail page is not a different product mode. It is the next level down from the overview. The current jump weakens orientation and makes the overview/detail pair harder to learn.

Fix:

- Create a shared `entity-header` or `detail-hero` component based on `section-hero`, not on `.panel.*-header`.
- Keep detail pages distinct through density and metadata, not through a full saturated background.
- Recommended shape for quality details: paper/tinted-blue surface, 5px quality-blue rail, H1, compact tags, "Characteristic" backlink pill, optional aliases row.
- Preserve category colors as rails, badges, and metadata accents. Avoid full-surface cyan/red/green/yellow except in small chips or special callouts.
- Apply the same family to quality, standard, approach, and possibly requirement headers. Requirements can keep a stronger red accent, but not a page-sized warning slab.

Primary files:

- `_includes/quality-header.liquid`
- `_includes/requirements-header.liquid`
- `_includes/standards-header.liquid`
- `_includes/approach-header.liquid`
- `_sass/components/_panels.scss`
- `_sass/components/_section-hero.scss`

Acceptance:

- `/qualities/` and `/qualities/accessibility` look related within one second.
- Detail headers remain visually different from index heroes but share border radius, rail grammar, spacing rhythm, and metadata row behavior.
- Category color is recognizable without dominating the reading surface.

### P1 - Rebuild tag pages as blue/navy dimension hubs

Problem: tag pages use `section="tag"` with a dark-blue rail, but the body is still a legacy document plus raw lists. The dark-blue signal is too small, and the page does not clearly say "this is a quality dimension hub".

Why it matters: dimensions are a major organizing concept in the metamodel. For dev teams, a dimension page should answer "what does this dimension mean, and what can I use now?" The current pages answer this only after scrolling and scanning.

Fix:

- Rename the visual role from `tag` to `dimension` in the hero API, even if URLs stay `/tag-*`.
- Use a darkish blue rail plus a subtle blue-tinted surface, for example navy rail with very light blue paper. This matches the metamodel without turning the page into the same cyan as qualities.
- Title pattern: `Quality Dimension: Reliable` or `#reliable` plus a visible small label "Quality dimension". Avoid a naked `#reliable` as the only H1.
- Add a short lede from the page content into the hero where available.
- After the definition/criteria block, show a compact inventory section with tabs or grouped panels: Qualities, Requirements, Approaches, Standards.
- Replace raw `ul.posts` lists with the same dense row component used by search or explorer results.
- Remove repeated `id="search-results"` from all include blocks.

Primary files:

- `_includes/dimension-header.liquid`
- `_pages/tag-*.md`
- `_includes/one-quality.liquid`
- `_includes/one-requirement.liquid`
- `_includes/one-approach.liquid`
- `_includes/one-standard.liquid`
- `_sass/components/_tag-chips.scss`
- `_sass/components/_section-hero.scss`

Acceptance:

- Tag pages read more blue/navy than neutral paper, but not as saturated as quality detail pages currently do.
- Users can jump to qualities, requirements, approaches, or standards without scrolling through every preceding block.
- Long dimension pages like `/tag-reliable/` and `/tag-usable/` no longer become plain link dumps.

### P1 - Make autocomplete results easier to discriminate

Problem: search is functionally strong, but the desktop result panel truncates exactly the titles a developer must compare. In the live `access` search, "Access Control via SSO", "Accessible User Interface", and "Access control is enforced" become visually compressed while paths occupy a second column.

Why it matters: autocomplete is now the fastest expert path through the site. If titles truncate, users slow down and fall back to scanning.

Fix:

- Increase desktop panel width from `max-width: 38rem` toward 46-52rem, or make the panel detach to a centered command-palette width on wide screens.
- Put path on a second line or hide it until hover/focus. Title must win over path.
- Add compact type labels or group rail + section label, but do not make paths compete with titles.
- Index dimension pages as first-class results, or at least add a "Dimensions" group when the query matches tags.
- Consider a top result rule for exact title matches, especially for qualities and standards.

Primary files:

- `_sass/components/_search-autocomplete.scss:49`
- `_sass/components/_search-autocomplete.scss:104`
- `_sass/components/_search-autocomplete.scss:152`
- `src/scripts/site/autocomplete.js`
- `src/scripts/index-search.js`

Acceptance:

- "Accessible User Interface" and similar long result titles are readable at desktop width.
- `access`, `reliab`, `wcag`, `iso 25010`, and `operable` produce immediately interpretable grouped results.
- Dimension pages can be reached through search as quickly as qualities.

### P2 - Replace saturated related-section bars on detail pages

Problem: lower sections on quality pages use full-width saturated bars for "Related Approaches" and "Related Standards". On `/qualities/accessibility`, these bars compete with the content more than they structure it.

Why it matters: related entities are navigation aids, not primary page headers. The current bars make every relation feel like a new top-level page.

Fix:

- Convert `h2.section-heading` into a quiet section header: icon, title, count, optional colored rail or small badge.
- Use light tinted backgrounds only when a section contains a grouped list or dense panel.
- Keep entity colors, but reduce them to rails, icons, count pills, or small label backgrounds.

Primary files:

- `_sass/components/_section-headings.scss`
- `_includes/directly-related-qualities.html`
- `_includes/directly-related-requirements.html`
- `_includes/directly-related-approaches.html`
- `_includes/directly-related-standards.html`

Acceptance:

- Related sections are visibly entity-specific but do not overpower definitions.
- A quality page scans as one article with supporting links, not as four stacked colored landing pages.

### P2 - Promote dimensions in navigation or label them more explicitly

Problem: dimensions are currently secondary navigation (`_data/navigation.yml:13`) while qualities, requirements, approaches, and standards are primary. That is reasonable if dimensions are just filters. It is less reasonable if tag pages are the blue "heading" layer for qualities.

Why it matters: users need to understand the hierarchy: dimensions organize qualities; qualities connect requirements, approaches, and standards. Hiding dimensions makes the metamodel harder to learn.

Fix:

- Either promote "Dimensions" into primary nav, or add a dimensions link beside the `/qualities/` filters and tag page hero.
- Rename "How-To" to "Start here" if it stays primary. "How-To" feels like internal docs wording.
- Keep the hamburger for lower-frequency resources: Aliases, Articles, References, About, Contact.

Primary files:

- `_data/navigation.yml`
- `_includes/site-header.html`
- `_pages/10-quality-dimensions.md`
- `_pages/20-quality-characteristics.md`

Acceptance:

- A first-time architect can discover dimensions without opening the hamburger.
- The nav reflects the metamodel rather than only the content collections.

### P2 - Add stronger information-reliability cues

Problem: the site has strong sources, but source confidence is not consistently surfaced. Some source links are buried in prose or blockquotes. Related standards counts are useful but not always presented as evidence.

Why it matters: dev teams use this during real decisions. They need to know whether a definition comes from ISO, Wikipedia, Bass, site authors, or a generated/example requirement.

Fix:

- Standardize source blocks: label, source name, version/year, and link.
- For quality and tag pages, distinguish "definition", "note", "source", and "site commentary" visually.
- On detail headers, consider a compact "Standards: N" or "Sources: ISO 25010, WCAG" line when present.
- Avoid vague sentences like "There is a W3C standard on this topic" when the exact standard can be linked and named.

Primary files:

- `_layouts/qualities.html`
- `_includes/quality-header.liquid`
- `_sass/_content.scss`
- content files under `_qualities/` and `_pages/tag-*.md`

Acceptance:

- A reader can tell at a glance whether a definition is normative, encyclopedic, or project-authored.
- Standards evidence is visible before the footer relation lists.

### P3 - Clean implementation details that create visual rough edges

These are small individually, but they compound:

- Inline styles in tag include blocks: `_includes/one-quality.liquid:10`, `_includes/one-requirement.liquid:5`.
- Repeated `id="search-results"` on tag pages.
- Hover scale on tags, `transform: scale(1.25)`, can feel jumpy and may disturb dense reading: `_sass/components/_tag-chips.scss:15`.
- Negative letter spacing appears in several new heading rules. For this product surface, use the typefaces at their natural spacing unless a real rendering issue appears.
- `DESIGN.md` is missing. The redesign now has enough conventions to document: hero grammar, entity colors, rails, badges, panels, search, relation lists, and source blocks.

Acceptance:

- Components can be reused without inline exceptions.
- Dense pages do not shift or jump on hover.
- Future visual reviews can compare against a documented design vocabulary instead of inferring intent from scattered CSS.

## Persona red flags

Alex, power-user architect:

- Search is now the correct primary tool, but result truncation slows expert comparison.
- Dimensions are not primary, even though power users often start from `reliable`, `secure`, `usable`, or `operable`.
- Tag pages make Alex scroll through long explanatory content before reaching inventory.

Sam, accessibility-dependent user:

- Search combobox structure is promising. It still needs screen-reader verification for active option announcements and panel updates.
- Full-color slabs and hover-scaling chips create avoidable visual load.
- Long tag-page lists need better headings and skip links within the page.

Project-specific persona, senior architect in a workshop:

- They need to answer "which quality, requirement, and standard backs this decision?" quickly.
- The current information is present, but source confidence and relation hierarchy are not consistently visible at the point of decision.
- Header inconsistency makes it harder to teach the metamodel from the UI itself.

## Questions before implementation

1. Should tag pages be presented as "Quality Dimensions" even while keeping `/tag-*` URLs?
2. Should all entity detail pages move to the quiet detail-header family, or should requirements keep a more forceful red treatment?
3. Should autocomplete become the dominant navigation affordance, for example a wider command palette, or remain a compact header dropdown?
4. Is the current metamodel color scheme fixed as exact hex values, or can we introduce lighter/darker semantic variants for surfaces, rails, text, badges, and borders?

## Suggested implementation order

1. `detail-header-unification`: build the shared detail hero and migrate quality pages first.
2. `dimension-pages-blue-hubs`: redesign tag pages and the include blocks.
3. `search-result-readability`: widen/reflow autocomplete and add dimension results.
4. `related-section-headings`: replace saturated relation bars with quiet section headers.
5. `nav-dimensions`: promote or cross-link dimensions explicitly.
6. `source-reliability-cues`: standardize source/evidence presentation.
7. `design-documentation-polish`: write `DESIGN.md`, remove inline styles and repeated ids, tame hover scale and heading letter spacing.
