# /impeccable critique — arc42 Quality Model · 2026-05-15

Second-pass review after the violet redesign on branch `418-violet-redesign`. Two independent assessments (qualitative LLM review + deterministic source/live probe) over **https://q42.netlify.app** and the source at `_includes/`, `_layouts/`, `_sass/`.

Audience this critique is written for: **software architects, dev-teams, requirements engineers, trainers** using the site during workshops, design reviews, tender writing. Mandate: **information reliability and effectiveness**, not marketing polish.

What landed since the 2026-05-13 critique:
- Unified `section-hero` component (`_includes/section-hero.liquid`, `_sass/components/_section-hero.scss`) applied to every **index/landing** page.
- Requirement detail pages got the breadcrumb + curated siblings (commit af5a504).
- Autocomplete search via ⌘/Ctrl+K (commit 6e22eef).
- Hero vertical footprint compressed; duplicate `<h1>` risk suppressed on listings.

What did **not** land: the **detail pages** still live in the old design system. That's the dominant finding below.

## Anti-pattern verdict

**Not** AI-slop. The deterministic scan returned:

- **Zero** `background-clip: text` / gradient text.
- **Zero** `backdrop-filter` glassmorphism.
- **Zero** `border-left ≥ 3px` side-stripe accents on cards.
- **Zero** blue-tinted decorative shadows (`rgba(0,184,245,…)`).
- 4 total gradients across all of `_sass/` (3 in `_aliases.scss`, 1 in `_standards.scss`). Limited and intentional.
- 2 `transform: rotate` hits: one is the magnifying-glass icon (not decoration); one is the alias-hero shine (legitimate AI-slop residue — see item 12).

The single design-language failure is **the dual hero system**, and that's a half-migrated codebase, not a generative hallucination. Verdict: **clean on style, half-done on system**.

## What's working

- **`_includes/section-hero.liquid:32-36` is the right component.** A `data-section` attribute drives a CSS custom property, the rail colour matches the homepage legend, the typography is editorial-correct. The fix is to extend it, not to revise it.
- **`_layouts/requirements.html:5-24`** — the curated "Top 8 siblings" with rationale ("shares N qualities" / "via #tag") is the strongest editorial-density pattern on the site. Port the structure to qualities and approaches.
- **`_layouts/requirements.html:97-110`** — the breadcrumb implementation with `aria-current="page"` is correct and accessible. Use it as the template for item (5).
- **Typography**: Libre Caslon h1 at clamped 1.6–1.95rem with `font-weight: 500` and `letter-spacing: -0.01em` on the section-hero is exactly the "well-designed technical handbook" register the PRODUCT.md brief asks for. Match it on detail pages.
- **Token discipline on the brand-blue family** (`_variables.scss:64-67,137-140`): four well-named shades cover quality node colour, dark-blue heading text, hover, and surface-tint. The proposed tag-rail token swap (item 2) uses what's already there.

## Priority issues — extended diagnoses

The numbered list above is the actionable summary. Below are the three findings that drive most of the prioritization.

### [P0] Detail pages live in a different design system from index pages

Files: `_includes/quality-header.liquid:4`, `_includes/requirements-header.liquid:4`, `_includes/approach-header.liquid:4`, `_includes/article-header.html:11`, `_includes/standards-header.liquid:4` — all render `.panel.X-header` blocks with a saturated background. `_sass/components/_panels.scss:1-87` defines them. Six surface colours: cyan `#00b8f5`, pink `#ffb3b3`, green `#92ef80`, yellow `#ffc95c`, navy `#1a3a5c`, lavender `#e9d9e8`.

Effect: a developer navigates from `/qualities/` (paper card + 5px blue rail) to `/qualities/accessibility` (full-bleed cyan slab, dark navy heading, no rail) and the design language resets. Same content type, different visual grammar.

The user's framing is correct ("the similarity could be emphasized"). Path: collapse the panel-headers to call `section-hero.liquid` with a chips slot, keep the rail, delete the saturated fill. The detail page then carries the section's identity via the same 5px rail the index page used — that's the visual continuity. Add a `chips` row beneath the hero for tags, and a breadcrumb row above it.

**Command**: `$impeccable distill` (collapse the redundant header system), then `$impeccable polish`.

### [P0] Tag pages stack five competing header styles in one scroll

`/tag-usable/` opens with the quiet paper section-hero (navy rail). Then the body renders:
- `_includes/one-quality.liquid:10` → full cyan `.panel.quality-header` `<h2>` ("Qualities tagged with #usable") with inline `style="margin: 0; padding: 1rem"`.
- `_includes/one-requirement.liquid:5` → full pink `.panel.requirements-header` `<h2>` ("Requirements tagged with #usable") with the same inline style.
- `_includes/one-approach.liquid:10` → full green `.panel.approaches-header` `<h2>`.
- `_includes/one-standard.liquid:19` → full yellow `.panel.standards-header` `<h2>`.

That's **five header styles on one page**, only one of which belongs to the new system. The fix is the `<h2 class="section-subhead" data-section="qualities|requirements|approaches|standards" id="…">` pattern in item (3): one selector, one rail rule, four `data-section` variants, no inline styles, no saturated fills, anchor ids for the meta-count links in item (4).

**Command**: `$impeccable harmonize`, then `$impeccable polish`.

### [P1] Tag-page rail color is metamodel-incorrect

`_sass/components/_section-hero.scss:43-45` sets the tag-page rail to `var(--dimension-background-color)` = `#1a3a5c` (`_variables.scss:92`). That's pure navy with no blue-family identity. The user's framing — **tags are a "heading for qualities"** (the blue node type) — calls for a colour that reads as part of the blue family. The codebase already has the right token: `--brand-blue-dark: #0077ad` (`_variables.scss:65,138`). Same hue as `--brand-blue` (`#00b8f5`), darker, more authoritative.

Concrete change (one line in `_section-hero.scss`):

```scss
&[data-section="tag"] {
  --cat: var(--brand-blue-dark);
}
```

If you want one more shade for hover or for the chips, add `--brand-blue-deep: #003366` to `_variables.scss` (which is `$brand-blue-text` repurposed).

`--dimension-background-color: #1a3a5c` is then used only by `/dimensions/` (the taxonomy index), which is a different surface. No more token conflation.

**Command**: `$impeccable colorize`.

But `/qualities/accessibility` does not; if the architect is collecting requirements *by quality*, they have to use the browser back-button. Item (5) fixes.
- **Workshop facilitator (Gernot's primary user)**: `/tag-secure/` has 27+ qualities, 19+ requirements, 5+ standards. Hits a five-style scroll wall with no in-page navigation. Items (3) + (4) reduce this to four anchored sub-sections under a unified rail.

Inconsistency reads as "site doesn't take its own structure seriously."

Sticky header has no `scroll-margin-top` (item 9), so anchored navigation jumps clip the target heading under the header. The autocomplete shipped with the right ARIA (`combobox` / `listbox`) — verify in NVDA before declaring item 9 cosmetic.

## Provocative questions

1. **Why do detail-page panel headers exist at all?** The breadcrumb + chips + h1 pattern from `_layouts/requirements.html:97-110` is so much stronger than the cyan-fill `.panel.quality-header` that the right move may not be "port section-hero to detail pages" but "port the requirements detail-page header pattern to qualities, approaches, standards". The section-hero would then live on index/landing pages only, and detail pages would have a strictly typographic header. Two patterns, but each tied to a clear use case: hero-card for index, breadcrumb+h1+chips for detail. Worth a 30-minute decision before items (1) and (5) collide.
2. **Should `--dimension-background-color: #1a3a5c` survive?** Item (2) removes its largest consumer. After that, it differentiates `/dimensions/` from `/tag-*/`. Is that differentiation load-bearing? If the dimension index could be brand-violet-deep (matching the homepage legend) instead, you'd retire the token entirely.
3. **Is the saturated-fill defensible on standards detail?** If qualities, requirements, approaches detail pages migrate to section-hero, `_layouts/page_standard.html`'s yellow fill is the last hold-out. Pick a side: design system is "rails only", or "rails + saturated fills with a stated rule". Current state is "rails + accidentally-surviving fills".
4. **What's the right tag-page structure for a workshop?** The current scroll-wall (one tag → 62 qualities + 41 requirements + 2 approaches + 5 standards) is a workshop-room nightmare. Curated top-N (per item 6's pattern) with a "show all" expand might serve the audience better than alphabetical dumps.
