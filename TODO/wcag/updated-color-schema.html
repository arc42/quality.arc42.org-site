Proposals

Proposal A (Recommended): Text-first hardening

Keep all core background colors and make text tokens WCAG-safe.

  • Header text: #deefb7 -> #123f4e (4.60 on current green hero)
  • Quality chip text: use #003366 on #00b8f5 (5.51)
  • Synonym badge text: #005987 on #c7f0fd (6.23)
  • Requirements strong/soft: #ffb3b3/#8b0000 (5.89), #ffe0e0/#8b0000 (8.10)
  • Standards strong/soft: #ffc95c/#2c3e50 (7.20), #ffefc8/#2c3e50 (9.64)
  • Approaches strong/soft: #92ef80/#1b5e20 (5.58), #d9f7d3/#1b5e20 (6.82)
  • Muted ramp on white: #6f6f6f, #6b6b6b, #5e5e5e, #4d6f84

Proposal B: Keep light header text

Preserve the current light-on-green header look by darkening only the header background.

  • Header background: #5fb49c -> #357360
  • Keep header text #deefb7 (new ratio 4.54)
  • Add soft chip pairs for all domains: quality #dff4fd/#0f6ba8, requirement #ffe0e0/#8b0000, standard #ffefc8/#2c3e50, approach #d9f7d3/#1b5e20
  • Use same muted text ramp as Proposal A

Proposal C: Two-level semantic tokens

Introduce a maintainable token model: strong tokens for large surfaces, soft tokens for chips/metadata.

  • Per domain: *-strong-bg/text and *-soft-bg/text
  • Large headers/section bars -> strong
  • Chips, badges, counters, helper labels -> soft
  • Prevents repeated regressions on tiny text

Recommended sequence: implement Proposal A first for fast reduction of violations with minimal visual churn. Later, migrate to Proposal C if you want stricter long-term token governance.
